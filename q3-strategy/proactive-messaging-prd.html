PRD — PROACTIVE MESSAGING

Proactive Messaging via Chat

Send parents the first message inside chat — meeting them at the right moment with something useful — through a backend injection mechanism and a trigger framework built to scale.

Status: Draft Owner: Sarah Moore Contributors: Marcus (eng), Raul (eng), Phillip (eng), Marion (lifecycle), Stacy (design) Last updated: Jul 8, 2026 Bet: Q3 — Proactive Chat
SUMMARY

TL;DR

Today, parents only get help when they think to open chat. Proactive messaging flips that — we reach out first, inside the chat experience, at moments where a nudge is genuinely useful (e.g., a parent who's been active a week but never set a sleep sound). It also reaches the many users who never open chat on their own, bypassing the steep drop-off from all Hatch users to first chat. Q2 proved parents will engage with a proactive prompt, but the messages lived on the home screen and didn't carry into chat, so the experience felt fragmented.

v1 delivers two things together: (1) a backend injection mechanism that drops a proactive message into the parent's chat so it's one cohesive conversation, and (2) a scalable trigger framework that decides who gets what message, when — without inundating anyone. The first message format we'll test is a content card, building on its strong early engagement signal. Because tooling (most likely Braze) is a long-term commitment, a core requirement of v1 is validating that the chosen tool can support the full trigger framework, not just a one-off send.

GOALS & NON-GOALS

What we're solving for

GOALS
  • Deliver proactive messages inside chat as one cohesive conversation.
  • Stand up a reusable trigger framework (who / what / when) that scales beyond one message.
  • Ship a first, concrete test using a content card and learn from it.
  • Confirm our tooling choice supports the full framework before we commit.
  • Drive repeat chat engagement — laddering to the retention north star.
NON-GOALS (v1)
  • Optimizing across every what/who/when variable — that's post-v1.
  • Full library of message formats (NL prompts, tips) — content card first.
  • A dedicated logging agent or deep log integration.
  • Replacing lifecycle email/push — this complements them.
  • Redesigning the entire home screen — we align with Stacy's work, not own it.

WHAT WE'RE BUILDING (V1)

The mechanism, the framework, and a first test

v1 ships the injection mechanism and the trigger framework together, tested through a single first message — a content card.

01
Backend chat-injection mechanism
A message triggered from our lifecycle tool lands directly in the parent's chat, so tapping in shows the same message that prompted them — one conversation, not two surfaces. Status: Marcus has built an endpoint the tool can call; deployed to QA and in testing.
02
Scalable trigger framework
A repeatable way to define who receives which message and when — with frequency capping and arbitration so a parent who matches several triggers at once isn't inundated. v1 launches one trigger, but the framework is designed for many.
03
First test: a content card
Given the strong early engagement on content cards, the first proactive message is a content card. It's concrete to ship, measure, and learn from — and it connects two Q3 bets (proactive + content) into one flow.
THE V1 FLOW, END TO END
  1. Audience generation: Redshift produces a target user list from defined criteria (e.g., parents with no Favorite set up, one week in).
  2. Trigger & send: The lifecycle tool (Braze) sends the targeted message to that audience at the chosen time.
  3. Injection: Braze calls Marcus's endpoint, which injects the message into the parent's Baby Agent chat and returns a conversation ID.
  4. Storage & typing: The message is stored as a "proactive" type (not a user message), so it can be filtered, measured, and surfaced separately from organic chat.
  5. Surfacing: The parent sees a chat-native entry point on the home screen ("you have a new message") — moving away from banners.
  6. Reply: The parent taps in and can respond; the conversation continues in chat.
FRAMEWORK DIMENSIONS

Every trigger must define four things

The framework is built around four dimensions. A trigger isn't complete unless all four are specified — that's what makes it a repeatable, measurable system rather than a one-off send.

01 — WHO
Who gets the message
The audience. Defined from behavioral + profile criteria — activity level, content configured (e.g., no pink noise set), child age, subscription tier, device usage — built as Redshift lists / Braze segments.
02 — WHAT
What message they get
The content and format. v1 is a content card; the framework maps each trigger to a specific message and supports future formats (sleep tips, NL command prompts) without engineering rework per message.
03 — WHEN
The timing of the message
When it sends — onboarding windows, regression age bands, post-purchase, time-of-day — plus frequency capping and arbitration so a parent matching multiple triggers isn't inundated.
04 — ACTION
What they do with it
The intended response and where it goes: tap into chat, reply (and where the reply lands), or act on the message itself (save/play a content card, set a sound). Each trigger names the desired next step, and we instrument what actually happens.
WORKED EXAMPLE — ONE COMPLETE TRIGGER
WhoParents who haven't set up a Favorite yet, one week after activation.
WhatA content recommendation encouraging them to set up their first Favorite — delivered as a content card.
WhenAt the one-week mark (tenure-based), within frequency-cap limits.
ActionThe parent taps in, engages with the recommendation, and sets up the Favorite. Success = Favorite created.
SUPPRESSION & FREQUENCY GOVERNANCE

Don't inundate — a governance layer over Who & When

Being eligible for a message is not the same as receiving it. A governance layer sits across the Who and When dimensions and decides, per parent, whether a message actually sends. Getting this wrong is the fastest way to lose trust — so it's a launch requirement, not a later optimization.

FREQUENCY CAP
A hard ceiling per parent
A maximum number of proactive messages per parent per window — e.g., one message per week — regardless of how many triggers they qualify for.
ARBITRATION
When several apply, one wins
If a parent qualifies for multiple messages in the same window, a priority order picks the single message that sends. The rest are suppressed or queued — never all fired at once.
DISENGAGEMENT SUPPRESSION
Stop if they're not biting
If a parent receives several proactive messages (e.g., 3) and never engages, pause proactive messaging — assume fatigue rather than keep pushing.
GLOBAL EXCLUDES
Never-message states
Suppress regardless of eligibility when a parent has opted out, was very recently messaged, or is in an active consultant / human escalation.
DECISIONS TO MAKE
  • Priority order: When multiple messages apply, what ranks them? (Proposed: setup-criticality / funnel stage — e.g., "no Favorite yet" outranks a nice-to-have tip.)
  • Send-all vs. suppress: Do we send everything a parent is eligible for, or cap and hold the rest? (Proposed: cap + suppress to protect trust.)
  • Queue or drop: When the cap suppresses a message, does it wait for the next window or get dropped entirely?
  • Disengagement threshold: How many un-engaged messages before we pause (3?), how long is the cooldown, and do we ever resume?
FUNCTIONAL REQUIREMENTS Requirements — 13 items; expand for the engineering checklist
INJECTION MECHANISM
FR-1
The system must accept a trigger call from the lifecycle tool and inject a message into the specified parent's chat, returning a conversation ID.
FR-2
Injected messages must be stored with a distinct proactive type, distinguishable from user and assistant messages for filtering and analytics.
FR-3
A proactive message must render correctly in chat (including content-card format) and be visible when the parent enters the conversation.
FR-4
A parent's reply to a proactive message must be handled by Baby Agent.
TRIGGER FRAMEWORK — WHO / WHAT / WHEN / ACTION
FR-5 · WHO
Audiences must be definable from behavioral + profile criteria (activity, content configured, child age, subscription tier, device usage), built as Redshift lists / segments.
FR-6 · WHAT
Each trigger must map its audience to a specific message and format (v1: content card). New messages/formats must be addable without per-message engineering rework.
FR-7 · WHEN
Send timing must be configurable by segment (onboarding window, regression age band, post-purchase, time-of-day). (Frequency capping and arbitration are covered under Suppression & Frequency Governance below.)
FR-8 · ACTION
Each trigger must define the intended parent action / next step (tap in, reply, or act on the message — e.g., save or play a content card), specify where a reply routes, and instrument the actual outcome.
SUPPRESSION & FREQUENCY GOVERNANCE
FR-9
The framework must enforce a frequency cap per parent per window (e.g., one proactive message per week), regardless of how many triggers they qualify for.
FR-10
When a parent is eligible for multiple messages in the same window, arbitration must select a single message by priority order and suppress (or queue) the rest — a parent must not receive all eligible messages at once.
FR-11
Disengagement suppression: after a defined number of consecutive un-engaged proactive messages (e.g., 3), the system must pause proactive messaging to that parent for a cooldown period.
HOME-SCREEN ENTRY POINT
FR-12
Proactive messages must surface as a chat-native entry point on the home screen (e.g., "you have a new message"), not a banner — designed in alignment with Stacy's home-screen redesign.
MEASUREMENT
FR-13
The system must instrument injection success, tap rate, reply rate, action completion, and downstream return, with a holdout group for measuring lift.
TOOLING DECISION

The tool must support the full framework

KEY DECISION
v1 will likely run on Braze, and Marion's lifecycle team already has automation tooling built around it. Because this is a long-term commitment, we must confirm — before committing — that Braze (or any alternative) can support the entire trigger framework below, not just the single v1 send. A tool that only does one-off sends would force a re-platform the moment we scale.
